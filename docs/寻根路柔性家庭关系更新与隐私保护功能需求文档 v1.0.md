文档版本：v1.0  

对应系统版本：寻根路 v1.0  

前置依赖：族谱核心数据架构（persons、family_units、family_children）、用户系统、权限体系  

面向对象：开发团队、测试团队、家族管理员  

1. 功能概述

1.1 背景

现代社会家庭结构日益多元化，离婚率上升、再婚重组、单亲抚养、未婚生育等情况越来越普遍。传统族谱以“父系单线”为核心，难以适应这些变化。女性族人面临“出嫁—离异—再嫁或不嫁”等复杂人生轨迹，孩子在离婚后的跟随抚养关系也需要妥善记录。如果族谱不能如实反映这些现实，就会失去可信度和亲和力，导致族人抵触使用。

1.2 目标

• 允许族人（尤其是当事人）自主更新自己的家庭关系状态，无需经过繁琐的行政流程。

• 以亲和、温暖的方式引导用户更新，避免冰冷、评判性的语言。

• 严格保护隐私：关系变更的细节仅对当事人和家族管理员可见，其他族人仅看到更新后的结果。

• 保持族谱数据的连贯性和可追溯性，支持历史版本查看。

1.3 核心原则

1. 尊重当事人意愿：婚姻状态、子女归属由当事人自行决定如何记录。
2. 不评判、不歧视：界面语言中性温暖，不出现“离异”、“非婚生”等标签化词汇。
3. 隐私分级：变更记录仅当事人和管理员可见，族人仅看到当前有效状态。
4. 可追溯：所有变更保留历史版本，防止恶意篡改。

2. 角色与权限

角色 权限

本人 可更新自己的婚姻状态、子女归属、抚养关系；可查看自己的变更历史

家族管理员 可查看本家族所有成员的变更记录；可审核敏感变更（如涉及未成年子女归属）

其他族人 仅看到当前有效的家庭关系（如“配偶：王芳”），看不到变更历史和隐私细节

游客 不可查看任何家庭关系细节

3. 功能列表

3.1 家庭关系更新入口

3.1.1 触发位置

• 个人详情页：“家庭关系”区块增加“更新家庭状况”按钮。

• 用户中心 → 个人资料：增加“家庭关系维护”入口。

• 族谱树：右键点击自己节点，菜单中增加“更新我的家庭状况”。

3.1.2 更新范围

更新类型 说明

婚姻状态 已婚 / 离异 / 丧偶 / 再婚 / 未婚（可多次变更）

配偶信息 新增配偶、解除配偶关系（离异/丧偶）、更换配偶（再婚）

子女归属 确认子女跟随自己生活 / 不跟随自己生活 / 共同抚养

子女新增 新增子女（支持“未登记父亲/母亲信息”选项）

3.2 亲和交互设计

3.2.1 语言风格

• 不使用“离异”、“离婚”、“非婚生”、“私生子”等标签化词汇。

• 改用中性温暖的表达：

  • “婚姻状态变更” → “家庭状况更新”

  • “离异” → “目前未在婚姻关系中”

  • “非婚生子女” → “子女信息登记（父亲/母亲信息暂未录入）”

  • “解除配偶关系” → “更新配偶信息”

3.2.2 引导式流程

采用“叙述式”引导，而非“填表式”：

第一步：选择你要更新的内容
○ 我的婚姻状况有变化
○ 我的子女情况有变化
○ 我的配偶信息需要更新

第二步（以婚姻状况为例）：
请告诉我您目前的情况：
○ 我已婚，配偶是 [选择/输入姓名]
○ 我目前未在婚姻关系中
○ 我丧偶
○ 我再婚了，现在的配偶是 [选择/输入姓名]

第三步（若选择“未在婚姻关系中”）：
是否需要保留前任配偶的信息？
○ 保留在族谱中（作为历史记录，仅管理员和您可见）
○ 从族谱中移除（仅移除配偶关系，不影响子女信息）

第四步：确认并提交
您的信息已更新，感谢您帮助我们完善家族记录。


3.2.3 子女归属的柔性处理

当用户更新子女归属时：

关于 [子女姓名] 的抚养情况：
○ 孩子跟随我生活
○ 孩子不跟随我生活，但仍是家族成员
○ 我们共同抚养孩子
[注：无论选择哪一项，孩子在族谱中的位置不变，永远是这个家族的血脉。]


3.3 隐私保护机制

3.3.1 信息分级可见

信息类型 本人可见 管理员可见 其他族人可见

当前婚姻状态 ✅ ✅ ✅（仅显示“已婚/未婚”）

当前配偶姓名 ✅ ✅ ✅

历史婚姻记录 ✅ ✅ ❌

离异原因/时间 ✅ ✅（可选） ❌

子女归属细节 ✅ ✅ ❌（仅显示子女姓名）

子女是否亲生 ✅ ✅（可选） ❌

3.3.2 默认隐私设置

• 历史婚姻记录默认仅本人和管理员可见。

• 子女归属细节默认仅本人和管理员可见。

• 用户可在“隐私设置”中调整上述默认值。

3.3.3 特殊场景保护

• 女方离异后子女跟随对方：子女在女方族谱中仍保留，但标注“抚养关系已变更”（仅管理员可见）。

• 女方再嫁后与新配偶的子女：新子女可同时记录在女方和现任配偶的族谱中（需双方确认）。

• 未婚生育：子女信息可只登记母亲一方，父亲信息留空，标注“待补充”（仅管理员可见原因）。

3.4 管理员审核机制

3.4.1 自动通过 vs 人工审核

变更类型 审核方式

本人婚姻状态变更 自动通过（当事人自主权）

配偶信息变更（新增/解除） 自动通过

子女新增（本人为母亲） 自动通过

子女新增（本人为父亲，但母亲非本族成员） 需管理员审核

子女归属变更（涉及另一方异议） 需管理员审核

3.4.2 审核界面

• 管理员后台 → “族人管理” → “家庭关系变更审核”。

• 展示变更前后对比，标注变更人、变更时间。

• 管理员可选择：通过 / 驳回（填写理由）/ 标记为“需线下确认”。

3.5 数据追溯与版本管理

3.5.1 历史版本

• 每次家庭关系变更自动生成一条版本记录。

• 本人和管理员可查看历史版本时间线。

• 支持回滚到任意历史版本（需管理员确认）。

3.5.2 变更通知

• 关键变更（如子女归属变更、配偶解除）自动通知相关方（如另一方配偶、家族管理员）。

• 通知方式：站内信（默认），可选短信（需付费）。

4. 数据架构

4.1 新增表

-- 家庭关系变更记录表
CREATE TABLE family_relation_changes (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  person_id BIGINT NOT NULL,                 -- 变更主体
  change_type ENUM('marriage', 'spouse', 'child', 'custody') NOT NULL,
  previous_state JSON,                       -- 变更前的状态快照
  current_state JSON,                        -- 变更后的状态快照
  privacy_level ENUM('self', 'admin', 'clan') DEFAULT 'admin',
  change_reason VARCHAR(200),                -- 用户填写的变更原因（可选）
  approved_by BIGINT NULL,                   -- 审核管理员ID（NULL表示自动通过）
  status ENUM('pending', 'approved', 'rejected') DEFAULT 'approved',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 子女归属记录表
CREATE TABLE child_custody_records (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  child_id BIGINT NOT NULL,                  -- 子女ID
  parent_id BIGINT NOT NULL,                 -- 父/母ID
  custody_status ENUM('living_with', 'not_living_with', 'joint') DEFAULT 'joint',
  is_biological BOOLEAN DEFAULT TRUE,        -- 是否亲生（仅管理员和本人可见）
  effective_from DATE NOT NULL,
  effective_to DATE NULL,                    -- NULL表示当前有效
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 婚姻历史记录表（用于存储完整的婚姻时间线）
CREATE TABLE marriage_history (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  person_id BIGINT NOT NULL,
  spouse_id BIGINT NOT NULL,
  marriage_type ENUM('first', 'remarriage') DEFAULT 'first',
  start_date DATE,
  end_date DATE NULL,                        -- NULL表示当前婚姻
  end_reason VARCHAR(50),                    -- 'divorce', 'widowed', ''（当前婚姻）
  privacy_level ENUM('self', 'admin') DEFAULT 'admin',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);


4.2 索引建议

• family_relation_changes：索引 (person_id, created_at)。

• child_custody_records：索引 (child_id, parent_id, effective_to)。

• marriage_history：索引 (person_id, start_date)。

6. 错误处理与边界条件

场景 处理方式

用户试图删除最后一段婚姻记录（导致无配偶但子女仍在） 允许删除配偶关系，但提示“子女信息不受影响”

用户试图将子女从族谱中删除 禁止删除，提示“子女永远是家族血脉，不可删除。如需调整抚养关系，请使用‘子女归属’功能”

双方对子女归属有争议（如离异夫妇都声称孩子跟随自己） 标记为“争议中”，通知管理员介入，提供线下证明材料上传入口

用户频繁变更婚姻状态（如一个月内变更超过3次） 触发风控，后续变更需管理员审核

未成年人（<18岁）自行更新家庭关系 禁止，提示“请由监护人代为操作”

7. 非功能性需求

• 性能：家庭关系更新提交响应 < 1s；历史记录查询 < 2s。

• 安全：所有隐私字段在后端加密存储；前端仅展示当前用户有权查看的内容。

• 可扩展性：支持未来新增家庭关系类型（如同性伴侣、收养关系）。

8. 验收标准（DoD）

1. 用户进入“更新家庭状况”，选择“婚姻状况有变化”，填写新状态后提交，页面立即更新。
2. 用户选择“目前未在婚姻关系中”，前任配偶信息默认仅自己和管理员可见。
3. 其他族人查看该用户详情页时，仅看到当前配偶信息，看不到历史记录。
4. 用户新增子女时，可选择“父亲/母亲信息暂未录入”，子女正常显示在族谱中。
5. 管理员在后台可查看所有变更记录，并可审核需要人工处理的变更。
6. 用户可在“我的家庭关系变更历史”中查看所有历史记录，支持按时间筛选。

9. 与现有模块的关系

模块 关系

族谱核心数据 更新直接影响 family_units 和 family_children 表

个人详情页 展示更新后的家庭关系

族谱树 根据更新后的关系调整树形显示（如离异后配偶不再显示为当前配偶）

用户中心 提供“家庭关系维护”入口和“变更历史”查看

管理员后台 提供变更审核界面

隐私设置 用户可在设置中调整历史婚姻记录的可见范围

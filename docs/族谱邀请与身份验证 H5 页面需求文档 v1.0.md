族谱邀请与身份验证 H5 页面需求文档 v1.0

文档版本：v1.0  

对应系统版本：GeneaSphere Platform v1.0  

前置依赖：用户系统、家族系统、微信公众平台（服务号/订阅号）、二维码生成服务  

面向对象：开发团队、测试团队、家族管理员  

1. 功能概述

1.1 背景

家族管理员在修谱过程中，需要高效地将散落在各地的族人纳入族谱系统。传统方式（逐个录入、微信群发链接）效率低且容易出错。通过微信公众号 H5 页面生成专属二维码，族人扫码后即可自动加入族谱，并快速验证和完善个人信息。同时，族人之间也可以互相发送验证二维码，确保族谱数据的准确性和完整性。

1.2 目标

• 管理员可生成邀请二维码，族人扫码后自动加入对应家族，并进入身份验证流程。

• 族人加入后，可查看自己的信息（上下三代及旁支），确认或修改。

• 族人可生成验证二维码，发给其他疑似族人，对方扫码后可在 30 分钟内完成身份确认。

• 身份确认机制：通过知识问答（关于上下三代/旁支的问题）或熟人背书（由已确认的族人担保）两种方式。

1.3 用户故事

• 作为族长：我想生成一个二维码，发到家族微信群，大家扫码就能加入族谱，不用一个一个录入。

• 作为族人：我扫码后发现系统给我匹配的祖父名字不对，我可以修改并提交审核。

• 作为族人：我怀疑隔壁村的张三也是我们家族的，我发一个验证二维码给他，他扫码后回答问题就能确认身份。

2. 角色与权限

角色 权限

家族管理员 生成邀请二维码、管理审核请求、查看族人验证记录

已认证族人 生成验证二维码、参与身份背书、修改自己的信息

未认证扫码者 扫码后进入验证流程，验证通过后成为“已认证族人”

游客 不可扫码（需微信授权登录）

3. 功能列表

3.1 管理员生成邀请二维码

3.1.1 入口

• 家族管理员后台 → “族人管理” → “邀请族人” → 生成二维码。

3.1.2 二维码属性

• 每个二维码绑定一个家族（clan_id）和生成者（管理员）。

• 二维码有效期：7 天（可配置），过期后需重新生成。

• 二维码可下载为 PNG 图片，或复制短链接。

3.1.3 扫码效果

• 族人使用微信扫码后，进入 H5 页面（需授权获取微信头像、昵称、手机号）。

• 自动跳转至“身份验证”流程。

3.2 族人扫码后的身份验证流程

3.2.1 第一步：微信授权登录

• 用户点击“允许”获取微信基本信息（头像、昵称、手机号）。

• 系统自动检查该手机号是否已在家族成员库中：

  • 已存在：直接进入“信息确认”步骤（3.2.2）。

  • 不存在：进入“信息填报”步骤（3.2.3）。

3.2.2 信息确认（已有数据）

• 系统展示该族人在族谱中的现有信息：姓名、性别、出生年份、父亲/母亲/配偶/子女（上下三代）。

• 用户逐项确认：

  • 正确：标记为“已确认”。

  • 错误：点击修改，提交修改申请（需管理员审核）。

• 确认完毕后，用户正式加入家族，获得“已认证族人”身份。

3.2.3 信息填报（无数据）

• 用户填写：姓名、性别、出生年份、父亲姓名（选填）、母亲姓名（选填）、配偶姓名（选填）、子女姓名（选填）。

• 提交后，系统尝试自动匹配族谱中已有的人物（通过姓名、年龄、关联关系）。

• 匹配成功：自动关联并进入“信息确认”步骤。

• 匹配失败：进入“身份验证”步骤（3.2.4）。

3.2.4 身份验证（核心确认机制）

目标：在 30 分钟内确认扫码者确实是本族成员。

两种验证方式（可并行）：

方式 A：知识问答（自动）
• 系统根据族谱数据，随机生成 3 道选择题，每题 4 个选项。

• 题目来源：

  • 上下三代：如“你祖父的名字是？”、“你父亲的出生年份是？”

  • 旁支：如“你二伯的名字是？”、“你姑姑嫁到了哪个村？”

• 答对 2 题及以上即通过验证。

• 答错可重新抽题（最多 3 次机会）。

• 通过后自动加入家族。

方式 B：熟人背书（人工）
• 用户输入一位已认证族人的姓名或手机号，系统发送验证请求给对方。

• 对方收到微信通知（或短信），在 30 分钟内确认“此人是我家族成员”。

• 背书通过后，用户自动加入家族。

超时处理：30 分钟内未完成验证，二维码失效，需重新扫码或联系管理员手动添加。

3.3 族人互发验证二维码

3.3.1 生成验证二维码

• 已认证族人在“我的”页面 → “验证族人” → 生成二维码。

• 二维码包含：当前用户 ID、家族 ID、有效期（30 分钟）。

3.3.2 对方扫码流程

• 对方扫码后，同样进入 H5 页面，授权登录。

• 系统显示“你正在被 [张三] 验证为 [张氏家族] 成员”。

• 对方可选择：

  • 接受验证：进入 3.2.4 的身份验证流程（知识问答或熟人背书）。

  • 拒绝：放弃验证。

3.3.3 验证结果

• 验证通过后，双方均收到通知。

• 验证发起方可在“我的验证记录”中查看结果。

3.4 信息修改与审核

• 族人对自己的信息提出修改后，修改申请进入管理员后台。

• 管理员可对比修改前后的数据，并结合族谱逻辑判断合理性。

• 审核通过后，数据更新；驳回时填写理由。

4. 身份确认机制详细设计（核心）

4.1 知识问答题库生成规则

题目类型 数据来源 示例

直系长辈姓名 从 persons 表获取父亲、祖父、曾祖父 “你父亲叫什么名字？”

直系长辈生年 从 persons 表获取出生年份 “你祖父出生于哪一年？”

旁系亲属姓名 从 family_units 和 family_children 获取叔伯、姑姨 “你二叔叫什么名字？”

旁系亲属关系 从 family_units 获取配偶、子女 “你姑姑嫁给了谁？”

家族地点 从 settlements 或 migration_events 获取祖籍地 “我们家族的祖籍地是？”

难度分级：
• 初级：直系长辈姓名（正确率最高）

• 中级：直系长辈生年、旁系姓名

• 高级：旁系关系、祖籍地

出题策略：
• 优先选择该用户理论上应该知道的（如父亲姓名必定知道）。

• 若用户连续答错，降低难度。

4.2 熟人背书流程


[发起方] 输入对方姓名/手机号 → 系统查找已认证族人
  → 找到：发送微信模板消息/短信给被请求人
  → 被请求人点击链接，确认“是家族成员”或“不是”
  → 确认结果通知发起方
  → 若确认“是”，则发起方验证通过


安全限制：
• 每个已认证族人每月最多可为 5 人背书，防止滥用。

• 背书记录永久保存，可追溯。

4.3 30 分钟倒计时

• 从扫码进入 H5 页面开始计时。

• 页面顶部显示倒计时条。

• 超时后页面提示“验证超时，请重新扫码”。

• 超时后所有未完成的验证请求（知识问答、熟人背书）自动失效。

5. 数据架构

5.1 新增表

-- 邀请二维码记录表
CREATE TABLE invite_qrcodes (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  clan_id BIGINT NOT NULL,
  creator_id BIGINT NOT NULL,                -- 管理员用户ID
  code VARCHAR(64) UNIQUE NOT NULL,          -- 二维码唯一标识
  expire_at TIMESTAMP NOT NULL,              -- 过期时间
  scan_count INT DEFAULT 0,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 验证会话表（每次扫码创建一个会话）
CREATE TABLE verification_sessions (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  qrcode_id BIGINT NULL,                     -- 关联的二维码（可为空，如果是族人互发）
  inviter_id BIGINT NULL,                    -- 发起验证的族人ID（互发场景）
  scanner_openid VARCHAR(100) NOT NULL,      -- 扫码者的微信openid
  clan_id BIGINT NOT NULL,
  status ENUM('pending', 'passed', 'failed', 'expired') DEFAULT 'pending',
  verify_method ENUM('quiz', 'endorsement', 'auto_match') NULL,
  expire_at TIMESTAMP NOT NULL,              -- 30分钟截止时间
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 知识问答记录表
CREATE TABLE quiz_attempts (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  session_id BIGINT NOT NULL,
  question TEXT NOT NULL,
  options JSON NOT NULL,
  correct_answer VARCHAR(100) NOT NULL,
  user_answer VARCHAR(100) NULL,
  is_correct BOOLEAN NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 熟人背书记录表
CREATE TABLE endorsements (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  session_id BIGINT NOT NULL,
  endorser_id BIGINT NOT NULL,               -- 背书的已认证族人ID
  result ENUM('confirmed', 'rejected') NULL,
  responded_at TIMESTAMP NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);


5.2 索引建议

• invite_qrcodes：索引 (clan_id, expire_at)。

• verification_sessions：索引 (scanner_openid, status)。

6. 界面交互设计

6.1 H5 页面流程（扫码后）


扫码 → 微信授权登录 → 加载中...
  ├─ 已有数据 → 信息确认页（展示上下三代）→ 确认/修改 → 完成
  └─ 无数据 → 信息填报页 → 提交 → 自动匹配
       ├─ 匹配成功 → 信息确认页 → 完成
       └─ 匹配失败 → 身份验证页（30分钟倒计时）
            ├─ 知识问答（3题）→ 通过/失败
            └─ 熟人背书（输入姓名）→ 等待背书 → 通过/失败
            └─ 超时 → 提示重新扫码

7. 错误处理与边界条件

场景 处理方式

扫码时二维码已过期 提示“二维码已过期，请联系管理员获取新二维码”

扫码者微信未绑定手机号 提示“请先绑定手机号”

知识问答连续答错3次 提示“答题失败，请尝试熟人背书或联系管理员”

熟人背书请求发送后对方30分钟内未回应 自动标记为“超时”，可重新发起

信息修改提交后管理员长时间未审核 48小时后自动提醒管理员

扫码者已是该家族成员 提示“您已经是本家族成员”，跳转到家族首页

8. 非功能性需求

• 性能：二维码扫码后页面加载 < 2s，知识问答题目生成 < 1s。

• 安全：二维码需携带签名防止伪造；微信授权使用 OAuth 2.0；所有接口需校验身份。

• 兼容性：H5 页面需适配 iOS/Android 微信浏览器，以及 PC 端微信扫码。

9. 验收标准（DoD）

1. 管理员生成邀请二维码，有效期为7天，扫码后进入H5页面。
2. 已有数据的族人扫码后，可看到自己的信息并确认或修改，确认后自动加入家族。
3. 无数据的族人填写信息后，若系统匹配到人物，则进入确认页；若匹配失败，进入身份验证。
4. 知识问答：3题答对2题即通过，答错可重抽（最多3次）。
5. 熟人背书：输入已认证族人姓名，对方在30分钟内确认后，扫码者通过验证。
6. 30分钟超时后，页面提示并失效。
7. 族人互发验证二维码功能正常，扫码后进入同样的验证流程。

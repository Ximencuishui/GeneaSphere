根脉云谱（GeneaSphere）寻找小伙伴功能需求文档 v1.0

文档版本：v1.0  

对应系统版本：GeneaSphere Platform v1.0  

前置依赖：数据架构 v1.0（persons、media_archive、settlements）、迁徙地图模块、时光长廊模块  

面向对象：开发团队、测试团队、家族管理员  

1. 功能概述

1.1 背景

现代社会高速发展，城市化进程加快，许多人背井离乡，与童年时期的伙伴失去联系。传统的“寻人”方式依赖记忆和偶然，成功率极低。

然而，族谱平台天然具备解决这个问题的三要素：
• 时间：族谱记录了每个人的出生年代、成长年代。

• 地点（POI）：迁徙地图标注了每个家族成员生活过的村寨、城镇。

• 照片：时光长廊保存了大量老照片，记录了当时的场景和人物。

这三者结合起来，使得“寻找儿时伙伴”从大海捞针变成了有线索、有路径、有温度的精准匹配。

1.2 目标

• 利用族谱数据中的年代 + 地点 + 照片三重维度，帮助用户找到童年时期在同一地点生活过的同龄人。

• 不局限于本家族成员——可以跨家族匹配（因为儿时伙伴可能来自不同姓氏）。

• 匹配成功后，提供安全的联络方式（双向确认，保护隐私）。

• 整个过程融入“回忆”的情感体验，而非冰冷的数据库查询。

1.3 用户故事

• 作为中年人：我想知道小时候一起在村里池塘游泳的那些玩伴现在在哪。

• 作为老年人：我想找到当年一起上私塾/小学的同学，看看他们过得怎么样。

• 作为年轻人：我想看看爸妈小时候的朋友长什么样，听他们讲过去的故事。

2. 角色与权限

角色 权限

普通成员 可发起寻找小伙伴请求，查看匹配结果，发送打招呼消息

被匹配用户 可决定是否回应打招呼，是否公开联系方式

家族管理员 可查看本家族成员的发起的匹配请求（仅统计，不涉及具体内容）

游客 不可使用此功能

3. 功能列表

3.1 匹配逻辑（核心算法）

3.1.1 匹配维度

维度 数据来源 权重

地点重合 两人曾在同一个 settlement（村寨/街区）生活过 50%

年代重合 两人在同一地点生活的时间段有交集（年龄差≤5岁） 30%

照片交集 两人出现在同一张老照片中（通过 media_person_links 或人脸识别） 20%

3.1.2 匹配范围

• 本家族：自动匹配。

• 跨家族：需用户授权“允许跨家族寻找小伙伴”（默认开启，可关闭）。

• 公开数据：仅匹配已故人员或用户自愿公开的信息。

3.1.3 匹配结果排序

按综合得分从高到低排序：
• 照片交集 > 同村+同龄 > 同村+年龄差稍大 > 仅同村

3.2 寻找流程

3.2.1 发起寻找

入口：
• 个人详情页 → “寻找小伙伴”按钮。

• 时光长廊 → 某张老照片 → “照片里的人，你还记得吗？”。

• 迁徙地图 → 某个POI点 → “在这里生活过的人”。

操作：
1. 用户选择或确认自己的“童年所在地”（系统自动提取 birth_place 和 migration_events 中的地点，用户可手动补充）。
2. 选择童年时间段（系统根据出生年份自动推算 5-15 岁区间，用户可调整）。
3. 点击“开始寻找”。

3.2.2 匹配结果展示

• 卡片列表：展示匹配到的候选人，每张卡片包含：

  • 头像、姓名（可匿名，仅显示“张同学”）。

  • 匹配度（百分比）。

  • 匹配依据（“你们都在王家村生活过”“你们出现在同一张照片中”）。

  • 一张共同的回忆照片（如有）。

• 地图模式：在迁徙地图上高亮显示匹配到的伙伴的位置。

3.2.3 打招呼与回应

• 打招呼：用户可向匹配到的伙伴发送一条预设模板消息（如“你好，我也是王家村长大的，还记得我吗？”），可附带一张共同照片作为“信物”。

• 回应：对方收到站内信通知，可选择：

  • 接受：双方建立联系，可查看对方联系方式（需双方都同意公开）。

  • 婉拒：礼貌拒绝，不透露联系方式。

  • 忽略：不做任何操作。

• 隐私保护：在双方都同意之前，不暴露真实姓名、手机号、住址。

3.3 照片找人（特色功能）

3.3.1 基于照片的匹配

• 用户在时光长廊中打开一张老照片，点击“找找照片里的人”。

• 系统自动识别照片中的人物（通过 media_person_links 或 AI 人脸识别）。

• 对于已在本平台注册的人物，直接显示其当前状态（“张三，目前在广州”）。

• 对于未注册的人物，提示“暂未找到，建议分享照片到家族群”。

3.3.2 照片认领

• 用户看到一张老照片中有自己，可以“认领”自己。

• 认领后，系统将该照片关联到用户账号，并通知照片中其他已注册的人。

3.4 回忆地图（情感化展示）

3.4.1 童年足迹地图

• 在迁徙地图上，以用户童年所在地为中心，标注出：

  • 自己曾经生活过的地点。

  • 匹配到的伙伴们当前所在的位置（仅显示城市级别）。

• 点击伙伴的位置，可查看其“童年足迹”与自己的交集。

3.4.2 时光回溯动画

• 点击“回到童年”按钮，地图自动缩放到童年所在地，时间轴跳转到童年年代。

• 该地点的老照片以缩略图形式浮现，带有光晕效果（复用迁徙地图的图片展示组件）。

4. 数据架构

4.1 新增表

-- 小伙伴匹配记录表
CREATE TABLE buddy_match_records (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  requester_id BIGINT NOT NULL,              -- 发起人用户ID
  matched_user_id BIGINT NOT NULL,           -- 被匹配用户ID
  match_score DECIMAL(5,2) DEFAULT 0.00,     -- 匹配度（0-100）
  match_reasons JSON,                        -- 匹配依据（地点/年代/照片）
  shared_media_id BIGINT NULL,               -- 共同照片ID（如有）
  status ENUM('pending', 'accepted', 'declined', 'ignored') DEFAULT 'pending',
  contacted_at TIMESTAMP NULL,               -- 打招呼时间
  responded_at TIMESTAMP NULL,               -- 回应时间
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 童年地点表（用户自行补充的童年生活地）
CREATE TABLE childhood_places (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  user_id BIGINT NOT NULL,
  location_name VARCHAR(200) NOT NULL,
  lat DECIMAL(10,7),
  lng DECIMAL(10,7),
  start_age INT DEFAULT 5,                   -- 起始年龄
  end_age INT DEFAULT 15,                    -- 结束年龄
  period_description VARCHAR(200),           -- 如“小学时期”
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 照片认领记录表
CREATE TABLE photo_claim_records (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  media_id BIGINT NOT NULL,
  claimer_user_id BIGINT NOT NULL,
  position_description VARCHAR(100),         -- 如“左三”、“后排中间”
  verified_by_admin BIGINT NULL,             -- 审核管理员ID（可选）
  status ENUM('pending', 'approved', 'rejected') DEFAULT 'pending',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);


4.2 索引建议

• buddy_match_records：索引 (requester_id, status) 和 (matched_user_id, status)。

• childhood_places：索引 (user_id)。

• 用户可在“设置”中控制：

  • 是否允许被跨家族匹配到。

  • 是否公开童年地点。

  • 是否允许他人通过照片找到自己。

6. 错误处理与边界条件

场景 处理方式

无任何匹配结果 提示“暂时没有找到小伙伴，建议补充童年地点信息或上传更多老照片”

匹配到的用户已注销 自动过滤，不显示在结果中

打招呼后对方长期未回应（超过30天） 自动标记为“已过期”，释放匹配名额

用户拒绝匹配 30天内不再向该用户展示同一人

照片认领争议（多人认领同一个人） 提交给家族管理员审核，或根据照片上传时间判定

7. 非功能性需求

• 匹配速度：全平台匹配查询应在 3 秒内返回结果。

• 隐私安全：未经双方同意，不得暴露真实姓名、手机号、住址等敏感信息。

• 数据准确性：地点匹配使用 GeoHash 算法，确保“王家村”和“王家庄”等近似地名也能匹配。

• 可扩展性：匹配算法应支持后续接入更多维度（如同学校、同工作单位）。

8. 验收标准（DoD）

1. 用户设置童年地点为“王家村”，时间段 1985-1995，系统返回至少 1 个匹配结果（基于测试数据）。
2. 匹配结果展示匹配度和匹配依据，点击“打招呼”后对方收到站内信。
3. 对方接受后，双方可查看彼此选择的公开联系方式。
4. 用户在时光长廊打开一张包含多人的老照片，点击“找找照片里的人”，系统列出已注册的人员。
5. 用户可在设置中关闭“跨家族匹配”，关闭后不再显示其他家族的匹配结果。

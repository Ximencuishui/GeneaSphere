根脉云谱（GeneaSphere）归宗合并功能需求文档 v1.0

文档版本：v1.0  

对应系统版本：GeneaSphere Platform v1.0  

前置依赖：数据架构 v1.0（闭包表）、寻亲模块、家族管理员后台  

面向对象：开发团队、测试团队、家族管理员  

1. 功能概述

1.1 背景

寻亲成功后，外支系（申请方）需要正式并入主家族（接收方）的族谱树中。归宗合并是整个寻亲流程的终点，也是最关键的操作——它涉及两个独立家族树的数据融合，必须保证血缘关系的正确性、数据的完整性以及操作的可追溯性。

1.2 目标

• 提供安全、可控的支系合并工具，让管理员能直观地确认挂载点并执行合并。

• 合并过程中自动处理闭包表的重算，保证所有祖先/后代关系正确。

• 支持合并前的数据快照和合并后的回滚机制。

• 记录完整的合并日志，便于审计。

2. 触发条件

归宗合并功能仅在以下条件全部满足时可用：
1. 寻亲申请已被管理员初步审核通过（状态为“待合并”）。
2. 申请方已同意归宗（系统内确认）。
3. 双方家族均为平台内的有效家族（未被删除或冻结）。
4. 当前操作用户为主家族的管理员（Admin 或 Owner）。

3. 角色与权限

角色 权限

主家族管理员（Admin/Owner） 执行合并操作、指定挂载点、回滚合并

申请方代表（Editor 或更高） 仅可查看合并进度，不可操作

平台超级管理员 可强制合并或回滚（异常情况）

4. 功能流程

4.1 前置流程（在寻亲模块中完成）

1. 申请方发布寻亲帖 → 系统推荐匹配 → 申请方发起认亲请求。
2. 主家族管理员收到申请，查看比对信息，初步审核通过。
3. 双方进入“认亲确认”阶段，申请方需确认愿意归宗。
4. 状态变为“待合并”，此时归宗合并功能可用。

4.2 合并主流程

步骤一：进入合并界面

管理员在认亲申请详情页点击“执行合并”按钮，进入归宗合并向导。

步骤二：选择挂载点

界面：左侧展示申请方的支系树（简化版，仅显示关键人物），右侧展示主家族的全树（可缩放、搜索）。
操作：管理员在主家族树中选择一个人物节点作为挂载点。该节点将成为申请方支系的“新父节点”。
约束：
• 挂载点必须是主家族中的已故人员（避免在世人员被错误挂载）。

• 挂载点不能是申请方支系中已有的任何人（防止循环）。

• 系统自动校验挂载点的世代与申请方支系的世代是否合理（偏差不超过2代，否则警告）。

步骤三：确认世代对齐

系统根据申请方提供的字辈信息和挂载点的世代，自动推算申请方支系各成员的世代偏移量。
界面：显示一个表格，列出申请方支系的关键人物及其当前世代、合并后的新世代。管理员可手动微调偏移量（±1代）。
验证：系统自动检查调整后是否出现“父亲比儿子年轻”等逻辑错误。

步骤四：执行合并

管理员点击“确认合并”，系统执行以下操作（在单个数据库事务中完成）：
1. 创建快照：对主家族当前所有数据进行全量快照（存储为 JSON 或备份表）。
2. 更新人物关系：
   • 将申请方支系所有成员的 clan_id 更新为主家族的 clan_id。

   • 在 family_units 中创建一条新记录，将挂载点作为父亲/母亲，申请方支系的始祖作为子女。

   • 在 family_children 中添加对应记录。

3. 重算闭包表：
   • 为申请方支系所有成员重新计算 person_ancestry 记录，使其祖先路径包含主家族中挂载点以上的所有祖先。

   • 同时保留支系内部的原有祖先路径（深度相应增加）。

4. 更新寻亲状态：将认亲申请状态改为“已合并”。
5. 通知双方：发送系统通知给申请方代表和主家族管理员。

步骤五：合并结果展示

合并完成后，自动跳转到合并后的族谱树页面，高亮显示新挂载的支系。
管理员可在此页面检查数据正确性。

4.3 回滚机制

合并操作后24小时内，管理员可在“日志审计”中找到该合并记录，点击“回滚”按钮。
回滚逻辑：
• 从快照中恢复主家族的全部数据。

• 将申请方支系所有成员的 clan_id 恢复为原家族 ID。

• 删除合并时创建的 family_units 和 family_children 记录。

• 重算主家族闭包表（恢复到合并前状态）。

• 将认亲申请状态改为“已回滚”。

约束：回滚操作不可逆（即不能再次回滚回滚）。超过24小时或期间有其他重要操作（如新增成员），回滚功能将禁用，需联系平台管理员手动处理。

5. 数据变更详细说明

5.1 数据库表变更

新增记录

• family_units：插入一条记录，husband_id 或 wife_id 为挂载点，另一个为 NULL（单亲挂载），union_type 为 merger。

• family_children：插入记录，family_id 为新家庭单元ID，child_id 为申请方始祖ID。

• person_ancestry：为申请方所有成员新增祖先路径（从主家族挂载点以上的所有祖先到该成员的路径）。

更新记录

• persons：更新所有申请方成员的 clan_id 字段。

• seek_relative_posts：更新状态为 merged。

快照表（新增）

CREATE TABLE merge_snapshots (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  clan_id BIGINT NOT NULL,
  operator_id BIGINT NOT NULL,
  snapshot_data JSONB NOT NULL,  -- 全量数据快照
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  expired_at TIMESTAMP,          -- 过期时间（24小时后）
  is_reverted BOOLEAN DEFAULT FALSE
);


5.2 闭包表重算算法（伪代码）


function recalculateAncestryAfterMerge(mainClanId, branchRootId, anchorPersonId):
    // 1. 获取主家族中 anchorPerson 的所有祖先（包括自身）
    mainAncestors = SELECT ancestor_id FROM person_ancestry 
                    WHERE descendant_id = anchorPersonId AND clan_id = mainClanId
    
    // 2. 获取支系中 branchRoot 的所有后代（包括自身）
    branchDescendants = SELECT descendant_id FROM person_ancestry 
                        WHERE ancestor_id = branchRootId AND clan_id = oldClanId
    
    // 3. 对于每个支系后代，为其插入新的祖先路径
    for each descendant in branchDescendants:
        for each ancestor in mainAncestors:
            depth = (depth of ancestor in mainAncestors) + 
                    (original depth of descendant from branchRoot) + 1
            INSERT INTO person_ancestry (ancestor_id, descendant_id, depth, clan_id)
            VALUES (ancestor, descendant, depth, mainClanId)
    
    // 4. 更新支系内部原有的祖先路径的 clan_id 和 depth（depth 增加 offset）
    offset = max depth of mainAncestors where ancestor_id = anchorPersonId
    UPDATE person_ancestry SET 
        clan_id = mainClanId,
        depth = depth + offset
    WHERE clan_id = oldClanId AND descendant_id IN branchDescendants


6. 界面交互设计

6.1 合并向导页面（@/pages/Admin/MergeApplication.vue）

步骤条：挂载点选择 → 世代确认 → 执行合并 → 完成

第一步：挂载点选择

• 左侧面板：显示申请方支系的简化树（只显示始祖及以下3代）。

• 右侧面板：显示主家族树，支持搜索、缩放、点击选择节点。

• 选中节点后，显示该节点的基本信息（姓名、世代、生卒年）。

• 底部显示“已选择：XXX”字样，并有“下一步”按钮。

第二步：世代确认

• 表格展示申请方支系所有成员的原世代、系统推算的新世代。

• 管理员可对每个成员微调新世代（加减1），调整后系统自动校验父子关系是否合理。

• 若有冲突（如子代比父代年龄大），高亮显示并阻止继续。

第三步：执行合并

• 显示合并操作的概要信息（影响人数、挂载点）。

• 按钮：“执行合并”（需二次确认，输入管理员密码）。

• 执行过程中显示进度条（前端轮询后端状态）。

• 完成后自动跳转至合并后的族谱树。

6.2 合并记录查看（在日志审计中）

• 每条合并记录显示：时间、操作人、挂载点、申请方始祖、状态（成功/已回滚）。

• 点击“查看详情”可展示合并前后的数据对比（基于快照）。

• 24小时内且未回滚的记录显示“回滚”按钮。

7. 错误处理与边界条件

场景 处理方式

挂载点选择错误（如在世人员） 禁止选择，提示“请选择已故人员作为挂载点”

申请方支系与主家族存在重复人员（同名同生年） 警告并列出疑似重复人员，要求管理员确认是否合并或排除

合并过程中数据库异常（事务失败） 全部回滚，不产生脏数据，提示“合并失败，请重试”

合并后24小时内主家族有其他重要变更 回滚功能自动禁用（因为快照已不一致），提示“数据已变更，请联系平台客服”

申请方支系包含在世人员 允许合并，但合并后这些人员的隐私级别继承主家族的默认设置

挂载点本身无子女 正常合并，挂载点成为申请方始祖的父亲/母亲

8. 非功能性需求

• 性能：合并操作应在5秒内完成（对于支系小于1000人的场景）。超过1000人时，采用异步任务处理，前端显示进度。

• 数据一致性：合并操作必须在数据库事务中执行，任何一步失败则全部回滚。

• 安全性：合并接口需校验操作者的家族管理员身份，并记录操作日志。快照数据加密存储。

• 可恢复性：快照保留24小时，支持一键回滚。超过24小时需联系平台管理员从备份恢复。

9. 验收标准（DoD）

1. 管理员可选择一个主家族中的已故人员作为挂载点，执行合并后，申请方支系的所有成员出现在主家族树中，且祖先关系正确（向上追溯到主家族的始祖）。
2. 合并后，申请方支系成员的 clan_id 已更新，且其原有的内部关系保持不变。
3. 合并操作记录在日志中，24小时内可回滚，回滚后主家族恢复合并前状态，申请方支系恢复独立。
4. 在合并过程中发生网络中断或服务器重启，数据库保持一致性（无部分合并的数据）。
5. 对于包含在世人员的支系，合并后在世人员的隐私设置自动生效。

10. 与现有模块的依赖关系

模块 依赖关系

寻亲模块 提供认亲申请的状态流转，合并后更新状态

闭包表服务 核心依赖，合并时需调用闭包表重算接口

快照服务 合并前创建快照，回滚时恢复

通知服务 合并成功/失败/回滚时通知相关人员

日志服务 记录合并操作详情

权限服务 校验操作者身份，合并后更新申请方成员的权限角色

时光长廊照片标注功能需求文档 v1.0

文档版本：v1.0  

对应系统版本：寻根路 v1.0  

前置依赖：时光长廊模块（media_archive）、用户系统  

面向对象：开发团队、测试团队  

1. 功能概述

1.1 背景

时光长廊中的老照片承载着家族的记忆，但照片本身的信息往往有限——谁、什么时候、在哪里、发生了什么。仅仅依靠上传时填写的描述，无法充分挖掘照片背后的故事。用户需要一种轻量、直观的方式，对照片中的地点和时间段进行标注，并表达自己与那个地方的关系状态（生活过、玩过、听说过等）。这些标注不仅可以丰富照片的叙事层次，还能为“寻找小伙伴”、“迁徙地图”等功能提供更细腻的数据支撑。

1.2 目标

• 允许用户对时光长廊中的任意照片进行时间-地点-关系状态的三元组标注。

• 标注内容可被其他家族成员查看，并作为“共同回忆”的线索。

• 标注数据可被“寻找小伙伴”功能利用（例如：两个人标注了同一地点+同一时间段，即可产生匹配）。

1.3 用户故事

• 作为长辈：看到一张老宅照片，标注“1965-1975年在这里生活过”，让晚辈知道这是祖屋。

• 作为中年人：看到一张村口大树照片，标注“1978-1982年在这里玩过”，唤起童年记忆。

• 作为年轻人：看到一张陌生地点的照片，标注“听爷爷说过这个地方”，表达间接关联。

2. 角色与权限

角色 权限

普通成员 可标注任何本家族可见的照片；可查看所有标注

家族管理员 可删除不当标注

游客 不可标注，不可查看标注详情（仅可见标注数量）

3. 功能列表

3.1 标注操作

3.1.1 触发入口

• 时光长廊照片详情页（大图预览模式）底部增加“标注”按钮。

• 迁徙地图中弹出的照片缩略图下方增加“标注”入口。

3.1.2 标注表单

点击“标注”后弹出轻量表单，包含以下字段：

字段 类型 必填 说明

地点 文本输入 + 自动补全（基于已有POI库） 是 可输入村寨/街道/地标名称，系统从已有地点库匹配

开始年份 数字输入（1900-当前年份） 是 标注时间段的起始

结束年份 数字输入（≥开始年份） 是 标注时间段的结束

关系状态 单选 是 选项：生活过 / 玩过 / 听说过 / 路过 / 工作过 / 其他

备注 文本（最多100字） 否 补充说明

3.1.3 标注提交

• 提交后，标注立即显示在照片详情页的“标注列表”区域。

• 系统自动将地点解析为经纬度（如有），并关联到该照片。


3.2 标注展示

3.2.1 照片详情页

• 在照片描述下方，以时间线卡片形式展示所有标注。

• 每条标注显示：用户头像+姓名、关系状态图标、地点、时间段。

• 点击标注可跳转到该地点的迁徙地图视图。

3.2.2 个人中心

• 用户可在“我的标注”中查看自己所有历史标注，支持按照片、地点筛选。

3.3 标注的二次利用

3.3.1 寻找小伙伴增强

• 当两个用户对同一张照片或同一地点+重叠时间段进行了标注，系统自动提高他们在“寻找小伙伴”中的匹配度，并在匹配依据中显示：“你们都标注了王家村（1985-1990）”。

3.3.2 迁徙地图补充

• 标注中的地点可自动补充到迁徙地图的POI库（需管理员审核），丰富家族的活动轨迹。

4. 数据架构

4.1 新增表

-- 照片标注表
CREATE TABLE photo_annotations (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  media_id BIGINT NOT NULL,                    -- 关联的照片ID
  annotator_id BIGINT NOT NULL,                -- 标注用户ID
  location_name VARCHAR(200) NOT NULL,         -- 地点名称
  lat DECIMAL(10,7) NULL,                      -- 纬度（自动解析）
  lng DECIMAL(10,7) NULL,                      -- 经度
  start_year INT NOT NULL,                     -- 开始年份
  end_year INT NOT NULL,                       -- 结束年份
  relationship ENUM(
    'lived', 'played', 'heard_of', 'passed_by', 'worked', 'other'
  ) NOT NULL DEFAULT 'other',
  note VARCHAR(100) NULL,                      -- 备注
  is_verified BOOLEAN DEFAULT FALSE,           -- 是否已验证（管理员）
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 索引
CREATE INDEX idx_annotation_media ON photo_annotations(media_id);
CREATE INDEX idx_annotation_user ON photo_annotations(annotator_id);
CREATE INDEX idx_annotation_location ON photo_annotations(location_name, start_year, end_year);


4.2 现有表修改

• media_archive 表增加字段 annotation_count INT DEFAULT 0，用于快速显示标注数量，避免每次统计。

6. 错误处理与边界条件

场景 处理方式

地点输入为空 提示“请输入地点”

开始年份 > 结束年份 自动交换，或提示“结束年份不能早于开始年份”

地点解析失败（无法匹配到已知POI） 允许提交，但标注不会关联到迁徙地图，显示“未识别地点”

用户对同一张照片重复标注完全相同的地点+时间段 视为更新，覆盖原标注（保留历史版本，管理员可查看）

标注内容包含敏感词 自动屏蔽，提示“内容包含敏感词汇，请修改”

7. 非功能性需求

• 性能：标注提交响应 < 500ms；照片详情页标注列表加载 < 1s（分页，每页20条）。

• 数据量：单张照片标注上限 100 条，超出后隐藏较早标注，但仍可查询。

• 可扩展性：关系状态枚举支持后续新增（如“求学过”、“参军过”）。

8. 验收标准（DoD）

1. 用户在照片详情页点击“标注”，填写地点、年份、关系状态并提交，标注立即显示在列表中。
2. 标注显示用户头像、关系状态图标、地点、时间段、备注。
3. 同一用户对同一照片重复标注同一地点+时间段，原标注被更新。
4. 标注中的地点可被“寻找小伙伴”功能识别，作为匹配依据之一。
5. 管理员可在后台删除不当标注。

9. 与现有模块的关系

模块 关系

附录：标注数据与“寻找小伙伴”算法结合逻辑

本附录是对《寻找小伙伴功能需求文档 v1.0》中 3.1 匹配逻辑的细化补充，说明如何利用照片标注数据提升匹配的准确性和丰富度。

1. 标注数据在匹配中的角色

照片标注提供了三个关键维度：
• 地点：用户明确声明自己与某个地点有关联。

• 时间段：用户给出了在该地点的具体年份范围。

• 关系状态：用户表达了关联的性质（生活过、玩过、听说过等）。

这些数据比单纯的“出生地”更精细，也比迁徙地图中的“居住地”更主观、更贴近个人记忆。因此，它们是“寻找小伙伴”匹配算法的黄金信号。

2. 匹配分数的计算方式

2.1 基础分数（满分 100 分）

匹配维度 权重 说明

地点完全重合 40 分 两人标注的地点名称相同（或经 GeoHash 判定为同一地点）

时间段重叠 30 分 两人的标注时间段有交集（年份区间交叉）

关系状态互补 15 分 一人标注“生活过”，另一人标注“玩过”，说明可能是邻居/玩伴

同一张照片 15 分 两人标注了同一张照片（即使地点/时间不完全一致）
2.2 加分项（额外加分，不计入满分）
条件 加分 说明

两人标注的备注中提到相同的关键词（如“池塘”、“小学”） +5 分 说明有共同的记忆点

两人年龄差 ≤ 3 岁 +10 分 同龄人更可能是一起玩的伙伴

两人都标注了“生活过”或“玩过” +5 分 关系状态越接近，越可能是直接认识
2.3 降权/过滤规则
情况 处理

关系状态为“听说过”且无其他匹配维度 匹配度降至 30 分以下，不展示在结果前列

时间段重叠少于 1 年 匹配度减半

两人标注的地点虽不同但相距超过 10 公里 仅作为“弱匹配”展示，需用户手动确认

3. 匹配结果的展示增强

当标注数据参与匹配时，匹配依据的描述会更加生动：

原匹配依据：
“你们都在王家村生活过”

增强后的匹配依据：
“你在王家村标注了‘生活过’（1970-1985），他也标注了‘玩过’（1975-1980），你们可能在村口的池塘边相遇过。”

如果两人标注了同一张照片：
“你们都标注了这张‘1982年春节合影’，你站在后排左二，他在前排右一。”

4. 实现建议

4.1 查询逻辑（伪代码）

-- 查找与当前用户有标注交集的其他人
SELECT 
    pa2.annotator_id AS matched_user_id,
    COUNT(*) AS match_points,
    SUM(CASE WHEN pa1.location_name = pa2.location_name THEN 40 ELSE 0 END) +
    SUM(CASE WHEN overlap_years(pa1.start_year, pa1.end_year, pa2.start_year, pa2.end_year) THEN 30 ELSE 0 END) +
    ... AS total_score
FROM photo_annotations pa1
JOIN photo_annotations pa2 
    ON (pa1.location_name = pa2.location_name 
        OR pa1.media_id = pa2.media_id)
    AND pa1.annotator_id = :currentUserId
    AND pa2.annotator_id != :currentUserId
WHERE pa1.annotator_id = :currentUserId
GROUP BY pa2.annotator_id
ORDER BY total_score DESC
LIMIT 20;


4.2 实时更新策略

• 用户每次新增/修改标注后，异步触发一次“寻找小伙伴”匹配更新（延迟 5 分钟，避免频繁计算）。

• 匹配结果缓存 24 小时，用户主动刷新时重新计算。

4.3 隐私保护

• 标注数据仅对同家族成员可见（跨家族匹配时，只展示“有交集”的结论，不暴露具体标注内容）。

• 用户可在设置中选择“不允许我的标注用于匹配”，关闭后系统忽略其标注数据。

1. 功能概述

1.1 背景

家族成员之间有时需要就特定话题进行交流，例如：核对某位祖先的生平、商议祭祖活动安排、分享老照片背后的故事。现有的“个人详情页留言”功能过于碎片化，而微信群/QQ群又过于嘈杂且信息难以沉淀。因此，需要一个安静、有序、可沉淀的讨论空间。此外，讨论过程中产生的宝贵信息（如考证结论、活动决议）需要被记录下来，形成可查阅、可分享的成果文档。

1.2 目标

• 允许家族成员（可跨家族）创建主题讨论组，围绕特定话题进行异步交流。

• 强调“安静”：无实时推送、无弹窗提醒、无消息计数红点（仅通过“小组”入口内的未读数提示）。

• 支持文字、贴图（表情/图片）、外链分享，满足基本交流需求。

• 新增：支持对话题或小组讨论进行自动归纳，生成结构化的“讨论总结”文档（类似会议纪要），便于保存和交流成果。

• 讨论内容及总结文档可长期保存，便于后人查阅家族历史讨论。

2. 角色与权限

角色 权限

小组创建者 设置小组名称、简介、封面；邀请/移除成员；删除小组；置顶/删除话题；生成并管理讨论总结

小组成员 创建话题（帖子）、回复话题、上传图片/链接；举报违规内容；请求生成讨论总结（需创建者审批）

家族管理员 可查看本家族成员参与的所有小组；可强制移除本家族成员（不影响其他家族成员）

游客/非成员 不可见小组内容

3.4 讨论总结（新增）

3.4.1 触发生成

• 手动触发：创建者或获得授权的成员可在话题详情页或小组首页点击“生成讨论总结”按钮。

• 触发范围：

  • 话题级总结：仅针对单个话题的全部回复进行归纳。

  • 小组级总结：针对小组内一段时间（如最近7天、自定义时间段）的所有话题和回复进行归纳。

• 权限：默认仅创建者可生成小组级总结；话题级总结可由话题作者或创建者生成。

3.4.2 生成内容结构

生成的总结文档包含以下部分（自动填充）：
• 标题：自动生成，如“关于《XX祖先生平考证》的讨论总结”或“XX小组2026年6月讨论纪要”。

• 基本信息：小组名称、生成时间、参与成员列表。

• 讨论背景：提取话题首帖或小组简介中的核心问题。

• 主要观点：按话题或按回复时间线，归纳各方提出的论点、证据、疑问。

• 共识与分歧：标注达成一致的结论和仍存在的争议点。

• 待办事项：提取讨论中明确的行动计划（如“张三负责查找县志”、“李四联系宗亲会”）。

• 附件索引：列出讨论中涉及的图片、外链等参考资料。

3.4.3 生成方式

• AI 辅助生成：后端调用 LLM（如 GPT 或本地模型）对讨论内容进行摘要和结构化整理。

• 人工编辑：生成后的草稿支持创建者或授权成员在线编辑、调整措辞、增减内容。

• 版本管理：每次编辑保存为一个新版本，支持查看历史版本。

3.4.4 保存与分享

• 保存：总结文档保存在小组内，独立于话题列表，可通过“讨论总结”标签页查看。

• 导出：支持导出为 Markdown、PDF 格式。

• 分享：生成分享链接（仅小组成员可访问），或复制为纯文本粘贴到外部。

3.4.5 通知

• 总结生成完成后，向小组内所有成员发送一条站内信（标题：“讨论总结已生成：《XXX》”），不产生弹窗或声音。

• 成员可在小组入口看到一个小圆点提示（与未读话题共用）。

4. 数据架构

4.1 新增表（相对于 v1.0）

-- 讨论总结表
CREATE TABLE discussion_summaries (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  group_id BIGINT NOT NULL,
  summary_type ENUM('topic', 'group') NOT NULL,  -- 话题级/小组级
  source_topic_id BIGINT NULL,                   -- 话题级总结时关联的话题ID
  time_range_start TIMESTAMP NULL,               -- 小组级总结的时间范围起点
  time_range_end TIMESTAMP NULL,                 -- 小组级总结的时间范围终点
  title VARCHAR(100) NOT NULL,
  content JSONB NOT NULL,                        -- 结构化内容（包含背景、观点、共识、待办等）
  generated_by BIGINT NOT NULL,                  -- 生成者用户ID
  version INT DEFAULT 1,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  deleted_at TIMESTAMP NULL
);

-- 总结版本历史表
CREATE TABLE summary_versions (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  summary_id BIGINT NOT NULL,
  version INT NOT NULL,
  content JSONB NOT NULL,
  editor_id BIGINT NOT NULL,
  edited_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);


4.2 索引建议

• discussion_summaries：索引 (group_id, deleted_at) 用于小组总结列表查询。

5. 界面交互设计

5.1 入口位置

• 小组详情页：新增“讨论总结”标签页，位于“话题”标签页旁边。

• 话题详情页：在操作栏增加“生成总结”按钮（仅对授权成员可见）。

5.2 讨论总结列表页（@/pages/Groups/Summaries.vue）

• 卡片列表，显示标题、类型（话题/小组）、生成时间、版本号、参与人数。

• 点击卡片进入总结详情页。

5.3 总结详情页（@/pages/Groups/SummaryDetail.vue）

• 顶部：标题、生成时间、版本号、编辑按钮（权限控制）。

• 内容区：按结构化区块展示（背景、观点、共识、待办等），每个区块可折叠。

• 底部：导出按钮（MD/PDF）、分享链接按钮、历史版本列表。

5.4 生成流程交互

1. 用户点击“生成总结”按钮。
2. 弹出确认对话框，选择范围（话题/小组）和时间（仅小组级）。
3. 点击确认后，按钮变为“生成中…”（禁用状态），后台异步处理。
4. 生成完成后，用户收到站内信，同时总结列表中出现新条目。
5. 生成失败时，按钮恢复可用，并提示“生成失败，请重试”。

6. 安静机制详解

（同 v1.0，新增总结生成的通知遵循同样的安静原则：仅站内信，无弹窗/声音。）

7. 错误处理与边界条件

场景 处理方式

讨论内容太少（不足5条回复） 提示“讨论内容较少，生成的总结可能不够完整，是否继续？”

AI 生成超时或失败 返回错误信息，允许用户重试；提供“手动撰写”入口

编辑总结时多人同时编辑 采用乐观锁，后保存者提示冲突，需刷新后重试

导出 PDF 时内容过长 自动分页，确保每页内容完整

8. 非功能性需求

• 性能：AI 生成应在 30 秒内完成（对于 200 条回复以内的讨论）。超时转为异步任务。

• 存储：总结内容以 JSON 格式存储，单条不超过 1MB。

• 可扩展性：AI 模型可替换，通过接口抽象层解耦。

9. 验收标准（DoD）

1. 用户可在话题详情页点击“生成总结”，系统自动生成包含背景、观点、共识、待办的文档，并保存到小组的“讨论总结”标签页。
2. 生成后的总结支持在线编辑，编辑后版本号递增，历史版本可查看。
3. 总结可导出为 Markdown 和 PDF 格式，内容完整。
4. 生成完成后，小组成员收到站内信通知，小组入口出现未读圆点。
5. 对于少于 5 条回复的话题，生成前给出提示。
6. 总结内容中的待办事项可被单独提取并显示在小组首页的“待办”区域（可选）。

10. 与现有模块的关系

• AI 服务：新增 AI 摘要模块，可复用或对接第三方 API。

• 通知系统：复用站内信通道，新增通知类型 summary_generated。

• 导出服务：复用 PDF 生成模块（已有印刷模块的 Puppeteer 服务）。
